## 一、工坊排程:採納 A 案(獨立、可丟棄的研究切片)

- **性質**:系統研究 spike,不是第二章 M1、不是第一章回填;失敗可整包丟棄,不留相容性負債。**spike API 不構成正式相容承諾**。
- **硬邊界**(Sol 文件 §一,全數採納):獨立灰盒區(`spike_workshop/`);不引用第一章正式場景、不改 frozen assets;4 typed slots、2 槽各 2 替代件、2 校準項正/偏二態;**16 組裝狀態→僅 4 個 `apparatusProfile`(乾淨/系統偏移/隨機抖動/尾段模糊・混合)**;fixture 只讀 profile;無市場/金錢/尋物/跨章物品庫/正式存檔/角色對話/正式圖;素 UI 保三自由(替代件、裝配順序、校準時機);組好的裝置持續可見(作品感)。
- **九月基準不動**:學生試玩=第一章完整內容+完整美術、**無工坊**;工坊切片另場獨立測(完章後 8–10 分鐘或另組),數據另表,不與通關率混算。
- **診斷提示載體序**(憲章一落地):①裝置與數據先呈現異常 → ②旅人筆記中性觀察(不蓋舞台) → ③卡住/求助才由伽利略**問句**引導 → ④不用旅人心理 OS 替玩家下結論。探索=漸進自動;學者=預設原始資料+主動索取;兩模式物理結果與成功條件相同。
- **教學語言**:不用「上層/下層」,改「兩道設計、一道診斷」——比什麼?怎麼量?怎麼知道量得可信?共同核心=**改一件事,看結果是否只跟那件事走**。
- **通過門檻**(六條,Sol §七):90 秒無配方組出可運作裝置/70% 能歸因自己的選擇/三誤差中≥2 能先提出正確診斷測試/重玩換路徑/組裝時間<20%/16 態確定性重播+4 profile 視覺不矛盾。**擁有感不高於純配置、或診斷被提示取代 → 縮回配置選單,不因引擎已做而硬留。**
