## 一、裁決內容

1. **滿版舞台殼現在就搭**(不等 Sol 首批圖定版面)——正式呈現=逆轉裁判式:場景背景鋪滿、立繪站位、底部對話框+人物名牌。佔位色塊先行,Sol 的 runtime WebP 落地即自動換裝。
2. **逐字演出範圍:對話逐字、旁白與系統行整句淡入**。一律支援點擊跳完、再點翻頁;`prefers-reduced-motion` 全顯不動畫。

法源對齊:美術規格 v0.2 §2.1 本就把「對話框、人物名牌、選項、提示」列為全域資產;本裁決把呈現形式定案並提前實作表現層。**不改任何凍結基線**(劇本 v0.2.2/章規格 v0.1.1/灰盒規格 v0.1.2):引擎、資料、玩法、文字零變動,純表現層新增。

## 二、實作架構(雙殼一心)

```
chapter.html(灰盒殼,Sol 驗證基準,永不重構)──┐
                                             ├── chapter-ui.js(唯一遊戲 UI 邏輯)── narrative.js/engine.js/data(唯一真相)
stage.html(舞台殼,候選正式呈現)+ stage-ui.js ─┘
```

- **chapter-ui.js 只加了事件廣播**(`bd:line`/`bd:scene`/`bd:view`/`bd:start`,CustomEvent 單向發佈):chapter.html 無訂閱者=灰盒行為零差異。
- **stage-ui.js = 純表現層**:只訂閱事件+唯讀 scenes/assets;禁碰引擎與存檔(測試把關,見下)。打字機 30ms/字;換景淡入吃 tokens 的 `--motion-*`;立繪支援 ART-ADR-001 分層合成;玩家發言時對方立繪壓暗。
- **回顧抽屜**=原「紀錄流」降級安置(旅人筆記:證據列表+完整逐字稿),等於逆轉裁判的 backlog;螢幕閱讀器由 log 的 aria-live 播報一次,打字機視覺層 aria-hidden(不重複播報)。
- **存檔互通**:兩殼同一把 `bd_ch1_save`——同一份進度,雙殼隨時互換。
