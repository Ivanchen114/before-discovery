## 0. 當前狀態與凍結邊界

目前專案仍在 v0 灰盒審查階段。依架構計劃書的開發凍結令，**灰盒未通過前不得量產正式美術、影片、角色動畫或第二章素材**。

凍結期間允許：

- 撰寫與審核本規格。
- 建立資產命名、資料夾、技術接口與驗收模板。
- 蒐集可驗證的歷史參考資料與授權資訊。
- 以純灰盒建立版面、Canvas／SVG 實驗動畫與互動可行性測試。

凍結期間禁止：

- 批量生成角色、背景與 CG。
- 為尚未驗證的玩法製作精緻演出。
- 製作正式預告片或逐格角色動畫。
- 以美術完成度掩蓋核心玩法尚未通過試玩的問題。

**美術解凍條件**：灰盒通過既定門檻，總監以決策紀錄宣告進入 Art Proof／垂直切片美術階段。

## 1. 美術產品策略

### 1.1 建議定位

**歷史科學插畫 × 科普漫畫 × 2D 視覺小說 × 可操作的現代資料介面**。

歷史感主要由人物、建築、服裝、器材、材質與光線承擔；需要閱讀、比較與操作的 UI 維持現代、清楚、可存取。不得為了「古書感」犧牲文字辨識與數據判讀。

### 1.2 全系列視覺支柱

1. **證據比裝飾重要**：重要數據、變因與證據關係永遠高於背景細節。
2. **歷史可信但不做博物館蠟像**：時代、器材與服裝有可查來源；人物表情與戲劇構圖保留漫畫可讀性。
3. **玩家是認知主體**：動畫協助玩家看見操作造成的結果，不代替玩家宣布結論。
4. **跨章一致、章內有個性**：旅人筆記、證據卡、辯論與基本 UI 全系列一致；每章以時代色、材質與科學主題建立差異。
5. **可拆卸、可維護**：人物、表情、背景、道具、FX 與文字分層；禁止把對白或重要 UI 文字燒進點陣圖。

### 1.3 暫定色彩架構

具體色值待 Style Frame 定案；程式先使用語意 token，不直接依章節寫死色碼。

- `color-surface-world`：歷史場景底色。
- `color-surface-notebook`：旅人筆記底色。
- `color-ink-primary`／`secondary`：文字與線稿。
- `color-evidence`：可用證據與資料關係。
- `color-warning`：變因不受控、證據不足。
- `color-success`：主張成立、支柱破裂。
- `color-chapter-accent`：每章唯一主題色。
- `color-focus`：鍵盤焦點，不能與章節色混用。

## 2. 跨章資產架構

### 2.1 全域資產（製作一次、持續維護）

- 旅人筆記介面與翻頁／新增註記動效。
- 證據卡框架、類型標記、取得與選取狀態。
- 實驗台共用外殼、變因控制元件、比較表與圖表樣式。
- 辯論廳共用介面、證詞、追問、出示證據與信念支柱效果。
- 對話框、人物名牌、選項、提示、系統訊息。
- 信譽、說服力、章節進度與存檔狀態。
- 時間跳躍、穿越、場景切換、證據取得、支柱破裂等共用 FX。
- 無障礙設定、降低動態、文字大小與高對比模式。
- 系列 Logo、啟動畫面、章節選單與製作人員名單。

全域資產不得被單一章節的裝飾需求破壞。章節差異以 token、外框裝飾、材質與背景層處理。

### 2.2 每章新增資產

每章依「章節美術附錄」新增：

- 時代／地區色票與材質。
- 主要人物、次要人物與服裝設定。
- 場景、時段與必要變體。
- 實驗器材、史料、證據物與代表性道具。
- 該章代表性實驗的互動視覺。
- 1–3 張關鍵敘事 CG（只有確實無法由場景＋立繪完成時才製作）。
- 該章專用 FX、章封面與史實頁插圖。

### 2.3 每章暫定產量預算

下表是規劃上限，不是最低消耗；第一章完成後依實際工時修訂。

| 類別 | 每章暫定量 | 備註 |
|---|---:|---|
| 主要角色 | 2–4 人 | 每人 1–3 姿勢、6–8 基本表情 |
| 次要角色 | 2–6 人 | 優先共用姿勢與低差分方案 |
| 背景主場景 | 6–10 張 | 日夜、天候與年代變體另計 2–5 張 |
| 證據／器材／道具 | 10–25 件 | 可局部特寫、可進證據卡 |
| 代表性互動實驗 | 1–2 組 | 以程式動畫為主，不做預錄影片 |
| 關鍵 CG | 1–3 張 | 僅用於不可替代的高潮或時間跨度 |
| 章節專用 FX | 1–2 組 | 其餘沿用全域 FX |

每章同時受以下 **runtime 硬預算** 約束；產量表是數量上限，不能用來豁免檔案量上限：

| 預算項 | v0.2 暫定上限 | 計入範圍 |
|---|---:|---|
| 每章 runtime 總量 | 25 MB | 該章背景、角色、道具、CG、FX、音訊與章專用資料 |
| 首次可互動畫面下載量 | 3 MB | 首場景必要影像、字體、全域 CSS／JS 與 UI 資產 |
| 最大單一 runtime 檔 | 2 MB | 任一影像、音訊或核准影片衍生檔 |

超標時依序處理：輸出正確顯示尺寸 → 調整壓縮 → 延遲載入非首場景資產 → 減少不必要變體。仍超標才可向總監提交例外，且須附低階裝置載入測試。三個數字於 Art Proof 取得實測後可修訂，但修訂必須改本節並記錄理由，不能由單章自行放寬。

### 2.4 跨時代一致性

未來章節可能跨越不同世紀、地區與物理主題。統一感由以下元素維持：

- 旅人筆記和證據卡是跨時代不變的視覺錨點。
- 人物採同一比例、線條密度與表情語法。
- 場景可隨時代改變媒材感，但透視、光影與角色合成規則一致。
- 每章只選一個主題色與一個代表性圖形語彙，不重新設計整套 UI。
- 歷史名詞、人物姓名與重要標籤由 HTML 文本顯示，不嵌入圖片。

## 3. 影像、動畫與影片政策

### 3.1 點陣影像適用範圍

可使用生成與編修影像製作：

- 角色概念、立繪、表情、場景、敘事 CG。
- 器材、手稿、信件與其他歷史道具插圖。
- 章封面、史實頁插圖、網站宣傳圖。
- 可分層使用的霧、光、塵埃與材質來源。

生成結果不得直接視為完成品。每件資產須經構圖、角色一致性、手部／器材結構、歷史正確性、邊緣、色彩與尺寸檢查。

### 3.2 程式原生視覺適用範圍

下列內容由 Claude 以 HTML／CSS／SVG／Canvas 實作，Sol 提供視覺規格並驗收：

- 所有文字、按鈕、表格、圖表與狀態顯示。
- 斜面、落球、水鐘、計時、比較、數據生成等互動實驗。
- 可依玩家選擇即時改變的物理與因果視覺。
- 鍵盤焦點、hover、active、disabled、loading、error、empty、completed 狀態。
- 需要精確對齊、響應式縮放或無障礙支援的元件。

### 3.3 動畫分級

**必要動畫**：

- 玩家操作的即時反饋。
- 實驗運動、計時與數據出現。
- 證據取得、選取、比較與主張成立。
- 辯論支柱裂解、失敗與返回實驗階段。
- 場景切換與時間跳躍。

**有限角色動畫**：

- 表情切換、眨眼、呼吸、輕微姿勢與鏡頭位移。
- 使用分層立繪、CSS transform、交叉淡化或少量 sprite frames。
- 不製作全對白嘴型同步；不以逐格動畫覆蓋全章。

**降低動態模式**：

- 必須支援 `prefers-reduced-motion`。
- 關鍵資訊不得只靠移動、閃爍或震動傳達。
- 強烈震動、快速縮放與粒子可停用或改為淡入淡出。

### 3.4 遊戲內影片

預設**不使用全螢幕預錄影片作為主線敘事或實驗機制**。理由：檔案大、難以修改與翻譯、不能回應玩家狀態，也不利於低階裝置與網頁載入。

只有符合以下條件才可提案使用：

1. 內容是不可互動的短高潮或跨年代蒙太奇。
2. 分層場景與程式動畫無法以合理成本達成。
3. 片段不承載唯一的玩法資訊，且可略過、可暫停、有字幕或文字替代。
4. WebM 為主並提供相容備援；片段長度與檔案預算另案核准。

第一章的時間跳躍與月球尾聲，優先使用分層插畫、鏡頭移動、光影與粒子完成，不預設製作影片。

### 3.5 宣傳影片

宣傳 teaser／trailer 排在垂直切片可玩、正式風格與核心素材穩定之後。建議：

- 首支 teaser：30–45 秒。
- 以實際遊戲畫面為主，概念圖不可冒充 gameplay。
- 需另列音樂、音效、旁白、字體與素材授權。
- 不因宣傳需求反向改動尚未驗證的核心玩法。

## 4. 美術製作流程

### 4.1 Art Proof（美術解凍後第一批）

只製作能回答視覺方向的最小集合：

1. 三組 Style Frame：歷史場景、實驗場景、辯論場景。
2. 一位同伴科學家角色母版與表情測試。
3. 一位懷疑者角色母版與表情測試。
4. 一張工作室背景與一組歷史器材。
5. 一張「世界＋旅人筆記＋互動 UI」合成稿。
6. 一段代表性實驗與數據產生的程式動效 proof。
7. **以其中一位角色**完成立繪差分 runtime 合成 A／B 測試，決定全案量產策略。

Style Frame 經總監選定後，才建立正式 Style Bible 與量產模板。

第 7 項只測一位角色、同一姿勢與至少三個表情，不在 Art Proof 量產全角色：

- **方案 A：整圖差分**——每個表情輸出一張完整透明立繪。
- **方案 B：分層差分**——共用身體層，臉部／必要局部表情以固定畫布疊合。
- 方案 B 的母版與 manifest 必須記錄 `canvasWidth`、`canvasHeight`、`anchorX`、`anchorY`、裁切框與 layer 順序；不同表情不得靠 runtime 猜測對齊。
- 兩案在同一顯示尺寸量測：人工校正工時、檔案總量、HTTP／file:// 載入、縮放接縫、背景合成、表情切換與維護步驟。
- 結果以 ART-ADR 記錄後才可量產。若方案 B 的對齊／修圖成本抵銷流量收益，退回方案 A，並受 §2.3 預算約束。

### 4.2 角色流程

研究包 → 輪廓與服裝方向 → 角色母版 → 正／側／三分之四檢查 → 表情母版 → 姿勢與差分 → 透明邊緣整理 → 場景合成測試 → 驗收。

要求：

- 同一人物的臉型、年齡、髮型、服裝結構與身高比例穩定。
- 表情差異必須在遊戲顯示尺寸仍可辨識。
- 手持器材與手勢須符合實際用途。
- 不把人物姓名或對白畫進立繪。

### 4.3 背景流程

史料／建築參考 → 場景功能草圖 → 角色站位與 UI 安全區 → 透視與光源 → 完稿 → 日夜／狀態變體 → Web 壓縮 → 合成測試。

背景必須服務互動：玩家需要比較數據或選器材時，背景對比與細節應主動退讓。

### 4.4 器材與證據流程

每件科學器材需有：

- 名稱與資產 ID。
- 年代、地區與來源紀錄。
- 結構與用途說明。
- 可操作部位與狀態列表。
- 全景、介面縮圖與必要特寫。
- 「史實／合理重建／戲劇改編」標籤。

### 4.5 章節製作順序

章節設計凍結 → 章節美術附錄 → 史料與器材查核 → 資產清單與估時 → Style Key → 角色／背景／道具 → 互動整合 → 章節視覺 QA → Art Lock。

禁止在章節證據鏈與互動規格未凍結前大量生產素材。

## 5. 技術交付規格

### 5.1 建議資料夾（repo 建立後）

```text
/art
  /source                 # 母版、分層檔、提示與參考紀錄
  /style                  # Style Bible、tokens.json、角色與場景規範
  /chapters/ch01          # 各章來源資產
  /chapters/ch02
/public/assets
  /global                 # tokens.css、assets.js/.json、全域 runtime 資產
  /chapters/ch01          # 各章 assets.js/.json 與 runtime 資產
/docs/art
  /references             # 史料、授權、來源與查核紀錄
```

### 5.2 命名

檔名使用穩定 ASCII ID，顯示名稱另走在地化資料：

```text
ch01_char_galileo_neutral_v01.webp
ch01_bg_pisa_study_night_v02.webp
ch01_prop_water_clock_detail_v01.webp
global_fx_evidence_acquired_v01.svg
```

格式：`章節_類別_主體_狀態_版本`。不得使用「final」「最新版」「修正版2」等不可追蹤名稱。

### 5.3 檔案格式

- 角色／背景 runtime：優先 WebP；需要無損 alpha 時用 PNG 或支援 alpha 的 WebP。
- UI icon、線條、簡圖與可縮放圖形：SVG。
- 原始母版：保留最高品質 PNG／分層來源檔；runtime 壓縮檔不可反向當母版。
- 影片：僅經核准後使用 WebM，另備相容格式。
- 文字與數字：HTML／CSS，不燒進圖片。

### 5.4 暫定尺寸

- 設計基準畫布：16:9、1920×1080。
- 背景母版：至少 2560×1440，保留左右裁切安全區。
- 角色立繪母版：長邊至少 2400 px，透明背景，保留完整肢體與裁切版。
- 敘事 CG：至少 2560×1440。
- 證據／器材特寫：長邊至少 1600 px。
- 實際輸出尺寸與壓縮品質須由瀏覽器載入測試決定，不直接載入所有母版。

上述尺寸於第一次 Art Proof 整合後凍結；若主要裝置策略改變，須更新本節而非各自猜測。

### 5.5 響應式與安全區

- v0 灰盒依規格以桌面為主；v1 前必須補行動裝置方案。
- 桌面 `>=1024px`：世界畫面與側／疊層筆記可並置。
- 緊湊版 `768–1023px`：一次只強調一個主要面板。
- 行動版 `<768px`：單欄、底部主要操作、禁止依賴 hover。
- 人物臉部、證據重點與操作目標不得放在易被裁切區。

### 5.6 動效 token（暫定）

- `motion-instant`：狀態確認，約 100–150ms。
- `motion-fast`：按鈕、選取、面板反饋，約 180–250ms。
- `motion-medium`：翻頁、證據取得、場景層切換，約 400–700ms。
- `motion-dramatic`：支柱破裂、時間跳躍，約 800–1400ms。

精確值由整合 proof 決定。所有動效須可被降低動態模式替代。

### 5.7 Token 單一真相來源

- `/art/style/tokens.json` 是色彩、排版、間距與動效 token 的**唯一人工編輯來源**；Style Bible 引用 token 名稱，不另抄一份值。
- 資產管線由 `tokens.json` 產生 `/public/assets/global/tokens.css` 的 CSS custom properties；`tokens.css` 是 runtime 衍生檔，禁止手改。
- 遊戲在 file:// 下直接載入 `tokens.css`，不於 runtime fetch JSON。
- 管線測試須驗證 JSON schema、token 名稱唯一、必要 token 齊全，並驗證生成結果未落後於來源。
- Claude 負責生成器與 CSS 接口；Sol 負責 token 語意、值與視覺驗收。任何名稱變更視為接口變更，須同步檢查使用處。

### 5.8 遊戲 runtime 與資產管線邊界

- **遊戲本體維持免建置**：clone／解壓後可直接開啟已提交的靜態檔，不要求玩家安裝 Node、bundler 或執行指令。
- **美術來源允許獨立資產腳本**：只用於輸出多尺寸 WebP、壓縮、產生 token CSS、產生資產清單與驗證鏡像；不接管遊戲程式 bundle。
- 所有 runtime 衍生檔必須提交 repo；CI／製作者可重跑腳本，玩家不需要管線。
- 禁止以資產處理為理由引入改變遊戲執行方式的 bundler。若日後確有需要，必須另提專案 ADR，說明效益、file:// 影響、回退方案與維護成本。
- G1 前由總監將上述邊界核准為正式 ART／技術 ADR；在此之前只可製作最小 proof 腳本，不擴張工具鏈。

### 5.9 資產清單與延遲載入

- 每個全域／章節資產清單以 `assets.json` 作規範鏡像，runtime 使用同內容的 `assets.js` 全域載體；兩者須有深度一致性測試。
- `assets.js` 至少記錄資產 ID、路徑、種類、尺寸、檔案量、首屏必要性、章節與可選 layer／anchor 資訊。
- runtime 禁止用 fetch／XHR 讀取資產清單；影像延遲載入採原生 `loading="lazy"` 或由清單建立 `Image`／`<img>`。
- 首屏必要資產不得標示 lazy；非當前場景資產不得在入口一次全載。
- `assets.js`、相對路徑與實際檔案存在性須在 file:// 與正式託管環境各驗一次。

## 6. 無障礙、效能與內容規範

### 6.1 無障礙

- 文字與重要圖示須達可讀對比，不以顏色作唯一訊息。
- 所有可操作元素具清楚 `focus-visible` 狀態與鍵盤順序。
- 圖表提供文字／表格等價資訊；歷史插圖提供 alt 文本。
- 觸控目標不得因裝飾縮小。
- 字級放大後不得遮住證據數值或主要操作。
- 動畫與影片均不可是取得關鍵概念的唯一方式。

### 6.2 效能

- 每章 runtime、首屏與最大單檔分別不得超過 §2.3 的 25 MB／3 MB／2 MB 暫定上限；字體、全域 UI 與首場景必要程式同樣計入首屏。
- 首屏只載入當前場景必要資產，其餘延遲載入。
- 背景與角色採合理尺寸的 runtime 版本，不直接載入母版。
- 同角色表情優先共用身體層或採可維護的差分策略。
- 粒子、模糊與大面積 blend 效果須在低階裝置測試。
- 每次 Art Proof／章節整合都記錄總下載量、首屏下載量、最大單檔、主要資產解碼尺寸與低階裝置結果；超標不得等到章末才發現。

### 6.3 在地化

- 不在圖片內生成中文、英文或公式標籤。
- UI 預留較長語言的伸縮空間，避免固定寬度依賴中文短字串。
- 人名、年代、物理符號與單位由內容資料供應。
- 宣傳圖若需要文字，保留無字底圖與各語言獨立版。

## 9. 驗收與 Art Lock

### 9.1 單件資產驗收

- 內容與用途符合資產清單。
- 尺寸、格式、命名、透明邊緣與壓縮正確。
- 角色／時代／器材與已核准 Style Bible 一致。
- 無明顯生成瑕疵、錯誤文字、浮水印或不合理結構。
- 已記錄來源、授權、提示與版本。
- 在實際遊戲尺寸與背景上測過，不只看原圖。

### 9.2 整合驗收

- default、hover、focus、active、selected、disabled、loading、error、empty、completed 狀態齊全。
- 文字長短、資料缺失、慢載入、低效能與降低動態皆有合理行為。
- 玩家能辨識可操作物、目前目標、證據狀態與下一步。
- 動畫不搶先揭曉結論，不讓成功看似由 NPC 或特效完成。
- 瀏覽器截圖與錄影留存於審核紀錄。

### 9.3 Art Lock 條件

1. 該章功能與文本狀態已凍結。
2. 全部必要資產完成並通過單件驗收。
3. 桌面與既定響應式尺寸整合通過。
4. 歷史／物理查核與授權無阻擋項。
5. 降低動態、鍵盤操作與文字放大測試通過。
6. 總監核准章節 Art Lock。

Art Lock 後只修 A 級錯誤與已核准的必要調整；新增演出須另開變更單。

## 附錄 C｜技術可行性檢查逐條回應

### B-1｜接受

採用每章 runtime ≤25 MB、首次可互動畫面 ≤3 MB、最大單檔 ≤2 MB 的暫定硬預算，寫入 §2.3 與 §6.2。字體與全域 UI／程式計入首屏；Art Proof 後可依實測修訂，但須在本文件留理由，不得由單章自行放寬。

### B-2｜接受

以 `/art/style/tokens.json` 為唯一人工編輯來源，由獨立資產管線產生 runtime `tokens.css`。Style Bible 與程式只引用 token 名稱；runtime 直接載 CSS，不 fetch JSON。接口與責任寫入 §5.7。

### B-3｜接受，延後量產策略裁決

不在 v0.2 預判全案必用分層。§4.1 明定 Art Proof 只用**一位角色、同一姿勢、至少三表情**比較整圖差分與身體＋臉部疊層；同步量測工時、檔案量、錨點／裁切、接縫與瀏覽器合成，結果以 ART-ADR 決定量產方案。

### B-4｜接受

§5.8 明定遊戲 runtime 永遠交付已生成的靜態檔、玩家免建置；美術來源可使用獨立腳本產生尺寸／壓縮／token／manifest 衍生檔，輸出必須提交。G1 前另請總監核准 ADR，防止資產腳本滑坡成遊戲 bundler。

### B-5｜接受

§5.9 採 `assets.js` runtime 全域載體＋`assets.json` 規範鏡像並做一致性測試；不以 fetch 載 manifest。延遲載入使用原生 `loading="lazy"` 或動態 img，且 file://／正式託管兩環境都列入驗收。
