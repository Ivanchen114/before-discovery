## 一、章節驗收句

玩家必須親手完成以下推理，NPC 不得提前代答：

1. 石頭放手前與船共同前進；放手不會自動清除已有的水平運動。
2. 近似直線穩速時，石頭相對船落在桅腳附近；相對岸邊則一面向前、一面下降。
3. 船加速時相對落點偏後，減速時偏前；因此「穩速」不是裝飾條件。
4. 船上與岸上兩張圖可以同時正確；必須先問「相對誰」。
5. 船桅實驗只排除「若地球運動，落石必落後」這項反對，不能直接證明地球運動。

刪掉玩家操作後，G1–G5 任一證據鏈不得自動成立。

第三章使用獨立純函式引擎 `engine3.js`；零 DOM、零 RNG。所有畫面只讀狀態，不參與判定。

  cabinResults: {

    steady: { drip:[], toss:[] }
  },                               // 每次重做皆追加 {id,state,test,offset,spread,near,day}
  predictions: { accelerating:null, decelerating:null },
  speedRuns: { accelerating:[], decelerating:[] }, // 每次重做追加 {id,state,offset,predicted,outcome,matched,day}
  overlay: {

  },
  publicDemo: { procedure:[], runs:0, complete:false },
  audit: { wind:false, acceleration:false, paths:false, boundary:false, overclaimTried:false },

### R-SHIP3-01 確定性 fixture

- 停船、門閂／剪繩三筆相對桅腳位移：`−0.06, +0.03, +0.04`（單位＝手掌寬）。
- 手放三筆：`−0.34, +0.29, −0.21`，只用來顯示釋放不穩；不得因多測而自動變乾淨。
- 搶跑／加速：`−0.72, −0.66, −0.75`（負＝桅後）。
- 穩速：`−0.04, +0.05, +0.02`。
- 減速：`+0.69, +0.63, +0.71`（正＝桅前）。
- 「桅腳附近」門檻：`|平均位移| ≤ 0.15` 且 `最大絕對位移 ≤ 0.20`。
- 船艙滴水／拋接 fixture：四種條件各有 3 筆固定微變異，依重做次數循環取用；零 RNG、可重現。首筆為停船滴水 `offset=.03,spread=.06`、停船拋接 `−.04,.09`、穩速滴水 `.05,.07`、穩速拋接 `−.03,.08`；皆為 `near=true`，不得宣稱絕對同點。重做用來檢查可重複性，不是取得 G2 的強制筆數。

### R-SHIP3-02 動作契約

| 動作 | 前置條件 | 結果／拒絕 |
|---|---|---|
| `setRelease(mode)` | mode 封閉列舉 | 更換方式清空尚未成立的基準 |
| `calibratePlumb()` | 無 | +1 天；重複校準 no-op |
| `runBaseline()` | 已校準、已選釋放 | +1 天；三筆乾淨紀錄後 baselineReady |
| `runMast(window)` | baselineReady | `depart/drumOnly` 產生 accelerating；`stable` 產生 steady；只寫原始紀錄，不授予 G1 |
| `assertG1(baselineIds,mastIds,concept)` | ≥3 乾淨基準＋≥3 穩速紀錄 | 正確概念 `steady-shares-motion` 才取得 G1；混入錯誤船況或資料不足皆拒絕 |
| `runCabin(state,test)` | G1 | dock/steady × drip/toss 四格；同一格可自由重做，每次 +1 天並追加原始紀錄，不授予 G2 |
| `assertG2(cells,concept)` | 四格各至少一筆 | 必須引用四種條件，且概念為 `steady-matches-dock` 才取得 G2；重做筆數不影響解鎖資格 |
| `setSpeedPrediction(accel,decel)` | G2 | 只鎖一次；正解 behind/front，但鎖定時不提示 |
| `runSpeedChange(kind)` | 已鎖預測 | 每次 +1 天並追加紀錄，可不限次重做；只寫加速／減速落點，不授予 G3 |
| `assertG3(kinds,concept)` | 加速與減速各至少一筆 | 必須引用兩種船況、累積平均仍為相反偏移，且概念為 `speed-change-breaks-shared-motion` 才取得 G3 |
| `inspectRecordBeat()` | G3 | 依序亮起 0→1→2→3 號鼓點；讀完四拍才可進時間配對，之後可重播 |
| `alignRecords(pair)` | G3 且四拍已讀完 | 只有同號鼓點配成同一時刻可通過 |
| `transformRecords(kind)` | aligned | 只有 subtractMast 通過；只完成轉換，不授予 G4 |
| `assertG4(records,concept)` | 轉換完成 | 必須引用 ship/shore 兩份同步紀錄，且概念為 `same-event-different-reference` 才取得 G4 |
| `runPublicStep(step)` | G4 | 四步依序：baseline→stable-window→no-push→repeat |
| `answerAudit(q,evidence)` | publicDemo.complete | 正確配對封蠟；錯誤留步、不扣信譽 |
| `setBoundary(choice)` | 三封蠟完成 | overclaim 回傳 repDelta=−1 並要求改答；honest 取得 G5 |

所有非法動作回傳 `{state:原物件,error:code}`，不得耗天、不得部分改寫。

### R-SHIP3-03 證據所有權

- G1–G4 的 `run*`／轉換動作只產生原始結果；證據一律由 `assertG*` 取得。
- 斷言介面固定顯示「引用哪些紀錄」與「這些紀錄支持什麼」，不得只給單一確認鈕。
- 正確概念選項位置分散，不形成固定答案序列。
- 已取得證據要保存 `claims.g*`，旅人筆記可回看來源、主張與適用邊界。
- fixture 是可重現的教學代理，不宣稱為 1640 年原始數據。

## 四、同步紙帶 R-REF3

資料固定為四個共同鼓點 `t=0,1,2,3`：

- 船／桅杆岸座標：`x=0,1,2,3`。
- 石頭岸座標：`x=0,1,2,3`，`y=0,1,4,9`。
- 船上紀錄：`xRel=0,0,0,0`，`y=0,1,4,9`。

UI 起始把兩張紙上下分開，不透明疊合：上方明標「岸上紀錄，尺固定在碼頭」，同時畫石頭與桅杆離岸位置；下方明標「船上紀錄，尺固定在桅杆（桅杆＝0）」，畫石頭離桅杆位置。玩家須依序讀 0–3 號鼓點，同一拍在上下兩張紙同時亮起。

讀完後玩家選鼓點配對；錯把終點或不同號鼓點當成同一時刻時，先顯示 `2 ≠ 3` 的可見後果，再提示時間不一致。正確配對後，玩家選「每一拍：石頭離岸 − 桅杆離岸」才得到船上讀值。UI 必須同時保留兩張原始紙帶與換算痕跡，不可用動畫偷換資料。

完成換算後可展開選讀數學：`x（石頭相對船）＝x（石頭相對岸）−x（船相對岸）`。它只替已完成的白話操作命名，不要求玩家代公式、計算或答題。

G4 前所有玩家可見字串禁用「參考系」；可用「相對誰」「參考物」。G4 完成後才在筆記與史實頁命名現代術語。

## 五、公開質詢 R-AUD3

終局不是血條辯論。畫面固定顯示三枚未封蠟質詢：

1. 風：正解 G2；G1 回覆「這正是甲板有風的那筆」。
2. 加速：正解 G3；G2 回覆「那只比較停船與穩速」。
3. 兩張路徑：正解 G4；單選任一紙帶皆拒絕。

每次玩家選證據前，卡片顯示「它能支持什麼／不能支持什麼」的白話摘要，但不標正解。錯誤不扣信譽，避免把證據配對變成背卡號。

### R-AUD3-UX 終局攻防呈現

- C3-1 的公開不是宣傳表演：船才回港，岸上已有人只傳「第一回落後」，也有人只傳「後來都在桅腳」。公開的目的，是讓每一個落點都必須連同船況、鼓點、岸標與釋放方式一起展示。
- C3-1 不得以五顆無上下文的流程按鈕呈現。艦長、商人、船員與槳手依序提出「基準、穩速窗口、放手機關、封存預測、重複性」五項具體質疑；玩家每公開一項程序，畫面須保留提問者或審查者的當場回應。
- C3-2 每一道卡片固定顯示：**誰在質疑 → 質疑的完整句子 → 玩家選擇的紀錄 → 對方如何接受或指出缺口**。不得只顯示 G1–G4 編號或「已完成」。
- C3-3 的官員是終局最後對手：他的過度宣稱要先以完整台詞出現，玩家再選擇順勢邀功或收住結論。
- 艦長的聲線可以強硬，但強硬必須指向責任與程序：他可叫停含糊條件、拒絕替學者背書，也必須在資料修正自己後承認泛化錯誤，並同樣叫停官員的過度宣稱。
- 這是公開證據審查，不複製前兩章教授法庭；但必須保有「對手施壓、玩家以證據答辯、對手回應、關係改變、問題升級」的戲劇節奏。
- 所有角色先說清物理因果，再用一句符合聲線的比喻收尾；比喻不得取代條件與原因。

最後邊界題：

- 選「證明地球在動」：信譽 −1、保存錯誤紀錄、強制回題，不重做公開驗證。
- 選「排除落石必落後的反對」：取得 G5，公開驗證完成。

### R-ART3-01 航船器材踏查

C1-1 進入實驗前顯示航船器材檢查圖，亮點固定為六項：釋放機構、鉛垂與桅腳標定、沙盤、鼓點與岸標、封閉船艙、同步紙帶。它是「理解每件器材控制什麼」的檢查，不是藏物遊戲：亮點清楚可見，全部點完才進核心實驗。固定骨架只說明責任，不提供假的更換選項。

世界圖負責器材情境；所有讀值、落點、紙帶與斷言來源由程式圖層呈現。停船、穩速、加速、減速、滴水、拋接與雙紙帶必須有不同可見狀態，不得只換文字、共用同一個冷冰冰圖示。

## 十、最低測試門檻

1. engine3 純函式、零 RNG、非法動作 identity 守衛。
2. 手放三次仍不能成立乾淨基準；門閂／剪繩可。
3. depart、drumOnly 皆不得作為 G1 來源；完成三筆 stable 仍不自動取得 G1，須正確選取三筆乾淨基準＋三筆 stable 並提出斷言。
4. 船艙四格少一格不得斷言；四格各至少一筆後仍不自動取得 G2，須引用四種條件並選對概念。完成過的格子不得鎖死：玩家可自由重做，每次追加原始紀錄並更新筆數、平均偏移與平均散布。
5. 加速→後、減速→前；兩者同押時拒絕鎖定；各至少一筆後仍須玩家斷言才取得 G3。兩種船況完成後按鈕不得鎖死，每次重做須追加紀錄、更新筆數與平均偏移。
6. 未逐拍讀完四個鼓點不得配對；紙帶錯鼓點、錯轉換均不得進入斷言；轉換完成後仍須引用船上／岸上兩份紀錄才取得 G4。
7. 三道質詢錯證據不扣信譽、不封蠟；正確配對才封。
8. overclaim −1 且不取得 G5；改答 honest 後才可完成。
9. `參考系` lint 在 G4 前負向變異必敗。
10. scenes3 JSON/JS 深等、17 場景全可達、黃金路徑可完章。
11. ch1/ch2 全測試保持綠；chapter rail 三章路由、存檔鍵隔離。
12. 桌機 1280×720、手機橫屏 844×390：實驗與公開質詢無遮擋、無水平捲動。
13. C1-1 器材踏查六項齊全，全部點完才能進核心實驗；讀檔恢復不重複阻擋。
14. 舊存檔缺 `cabinResults / claims` 時安全補值；舊版單筆船艙與單筆變速紀錄須遷移為陣列且仍可重做；惡意 concept／source／變速讀值必須被 sanitize 拒絕。
