# 《發現之前》玩家介面資料 ID 巡查（2026-07-22）

## 結論

本輪對第一至第三章、內部 QA 切片、書信碼匯入與無障礙文字做一次完整資料 ID 邊界巡查。修正原則不是刪除 ID，而是把它們留在資料、存檔、測試與開發文件中；玩家介面一律映射為自然語言。這樣同時保住舊存檔相容性與可讀性。

## 本輪抓到的問題

| 類型 | 曾出現的形式 | 為什麼是問題 | 修正方向 |
|---|---|---|---|
| 場景游標 | `B3-D｜論拋體之運動` | 玩家不知道 B3-D 是什麼；它只對作者有用 | 公開只顯示場景名稱；原 ID 留在游標 |
| 證據代碼 | `F3 桌緣彈射`、`取得 S5` | 把製作索引當成遊戲名詞 | 顯示證據全名；狀態仍以 F/G/S 鍵保存 |
| 證詞代碼 | `s1/s2/s3` | 資料庫欄位外洩 | 顯示「第一句／第二句／第三句證詞」 |
| 引擎拒絕理由 | `缺 E3.c` | 玩家知道自己被擋住，卻不知道少做了什麼 | 改成「尚未完成變換傾角的複驗」 |
| 製作流程標籤 | `死路 A`、`修復:` | 作者分類不是故事世界的語言 | 原 title 保留；公開顯示層移除前綴 |
| 匯入錯誤 | `schema/cursor/enum/profile/cycle` | 技術診斷不能指導玩家下一步 | 改為版本、故事位置、裝置狀態等白話 |
| 備援與無障礙文字 | `label || id`、asset id 當 alt | 正常路徑不出錯時難發現，一缺資料就外洩 | 玩家端備援只使用通用名稱 |
| 回歸測試缺口 | 規則只攔 `S1–S4` | 新增 `S5` 後測試仍全綠，形成假安全 | 擴充代碼範圍與來源級契約 |

## 邊界定義

### 必須保留的內部資料

- 場景、節點、證詞與證據 ID。
- 存檔章別、schema、游標與 localStorage key。
- 測試名稱、規格編號、ADR、資產 ID 與程式路由。
- 玩家產生的紀錄 ID 本身仍保留；其公開名稱是「實驗紀錄 #n」或「主張 #n」。

### 禁止直接出現的玩家表面

- 場景 HUD、章節標題列、對話與系統訊息。
- 證據卡、取得提示、辯論選項與實驗回饋。
- 進度匯入/匯出提示、檔名與所有錯誤訊息。
- `alt`、`aria-label`、`title`、`placeholder` 等無障礙/輔助文字。

## 根因

1. **資料與顯示共用同一欄位**，但缺少正式的 view-model 翻譯邊界。
2. **正常路徑有名稱，備援路徑偷懶回傳 ID**，造成低頻但真實的外洩。
3. **測試只列舉當時已有的代碼**，新增章節後沒有一起擴充。
4. **把內部 QA 當成不必可讀**，結果同一套壞語言容易被複製回正式介面。
5. **錯誤訊息由資料驗證器直接生成**，沒有再過玩家語言層。

## 永久防線

1. 所有場景名稱走 `TextFormat.playerSceneTitle`；不改原始場景 ID/title。
2. 證據、支柱、錯誤與資產替代文字都必須有自然語言映射；不得以 ID 作玩家端 fallback。
3. 自動測試同時掃描資料可見欄位與 UI 原始碼中的高風險組字模式。
4. 新章若新增證據字首或編號範圍，必須先更新測試正則，再接玩家介面。
5. 玩家版實機巡查仍不可省；機械掃描抓不到所有語境不自然問題。

## 出貨前五問

- 玩家看到這串字，能否只靠遊戲理解？
- 這是玩家需要引用的紀錄編號，還是作者定位用的 ID？
- 缺資料或載入失敗時，備援會不會吐出代碼？
- 螢幕閱讀器讀到的內容是否仍是人話？
- 新增了新的證據/場景命名範圍，測試是否同步擴充？
