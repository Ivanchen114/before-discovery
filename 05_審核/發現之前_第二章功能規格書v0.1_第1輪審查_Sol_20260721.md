## 一、判定

**不建議凍結；共 5 項 A 級、5 項 B 級、2 項 C 級。M1 暫不開工。**

規格的章骨架是對的：F1–F5、三支柱、說服力／信譽分流、敵方數據卡及三軌 lint 都接住了劇本 verification；第一章引擎沿用也比另立第二套安全。阻擋點集中在「規格還不能唯一決定程式行為」：工坊退回配置表、F2 沒有可執行 fixture、跨章存檔格式互相矛盾、FR 的操作順序與凍結劇本不同，以及 `√` lint 在揭曉邊界形成循環。

本輪實跑：

- `spike_workshop/tests.mjs`：**9/9 通過**；證明切片已有 typed slot、自由裝配順序、16 態→4 profile 與校準重置等可沿用結論。
- `greybox/npm test`：**66/66 通過**；只證明第一章現行 runtime 未受第二章紙上資產影響，不能替代第二章 fixture／跨章測試。

## 二、已確認通過

1. **R-DEB2-05 的意圖忠實**：敵方資料不是裝飾；玩家必須先讀斜率，再做「反駁／邊界／不足」分類；探索與學者的讀圖差分、a11y 數值替代及 FR 錯選不豁免都有寫入。
2. **失敗資源分流正確**：說服力歸零進 B3-F 配對複盤，信譽歸零才進 SC-R1；沒有重犯「辯論失敗卻回已完成實驗室」的問題。
3. **玩家不可替代節點已落規格**：F3 扳閂與判讀、F4 彎點標定都明定可錯可修，NPC 不可代答。
4. **戰利品版方向正確**：`拋物線` 的檢查範圍擴到 UI／選項／ARIA，接住 B-V3。
5. **跨章寫入邊界正確**：第二章不得回寫第一章存檔；無第一章存檔也不應形成玩法死路。

## 三、A 級｜阻擋凍結

### A-1｜名為「工坊首發」，實際規格仍是配置選單；切片三種受控自由全數遺失

**位置**：規格第 19–31 行；對照工坊裁決第 6–13 行、切片 README 第 12–23 行。

規格列出 `H／釋放刻度／球／桌沿／量法` 五個欄位，但沒有以下正式契約：

- typed slot 與零件 schema；
- 至少兩個槽的替代件；
- 自由裝配順序；
- 玩家自由決定何時校準；
- 換件後哪些校準失效；
- 組好後裝置持續可見；
- 不完整裝置是「不能運作」而非直接吐爛數據。

這些不是切片 API 細節，而是總監已裁的「Minecraft 擁有感」最低邊界。現稿可實作成五個 `<select>` 後仍宣稱工坊完成，驗收無法阻止。

**修正門檻（二擇一）**：

1. 正式化工坊：另立 `R-WS2`，定義 `parts／slots／calibrations／assemblyOrder／apparatusProfile`、place／replace／calibrate／run 的 Given-When-Then，並保留三自由與持續可見裝置；或
2. 明文撤回「工坊首發」，縮成配置介面。這是產品裁決，不能由實作者默默縮水。

### A-2｜R-LAB2 fixture 不足以產生唯一讀值或唯一判定

**位置**：規格第 25–32、75–78 行。

現稿只有理想值 `R=√H` 與四個 profile 名稱，缺少：

- 每個裝置狀態如何映射 profile（多個故障同時存在時誰支配）；
- 每個 profile × 輪替序號 × H 的確切讀值或確定性生成式；
- 未校準速度漂移是固定偏移、逐次抖動，還是隨 H 改變；
- 玩家必須選幾筆、是否可混球／混 profile、重複量測如何平均；
- `k` 如何由既有資料估計；
- 「√ 擬合偏差」的公式與第 25 格預測偏差的分母；
- 12% 通過、12.5% 不通過的安全邊際 fixture。

尤其固定的速度偏移只會改變 `k`，仍可能完美符合 `R=k√H`；若「未校準」要讓 F2 失敗，必須精確寫出它破壞的是重複性、比例形狀或外推，不能只貼故障名稱。

**修正門檻**：規格附 fixture 表／生成式與完整狀態枚舉，讓另一位實作者只讀規格即可重建相同讀值；用表中數據實跑通過格與失敗格後，才能決定沿用 12% 還是另訂閾值。閾值可在 M2 微調，但第一版可算 fixture 不可延後到 M2 才發明。

### A-3｜跨章旗標不存在，書信碼新格式也與第一章既有格式衝突

**位置**：規格第 15–17 行；現行 `greybox/src/narrative.js` 第 30–44、565–579 行與 `chapter-ui.js` 第 1–12、941–978 行。

規格讀取 `ch1.e3Certified`，但第一章 schema 3 沒有此欄位；現有事實來源是 `lab.evidence.e3.{a,b,c}`、章節 evidence 與 `ended`。規格又要求「書信碼內含章別欄」，但第一章 R-SAV-05 直接匯出 raw serialized state，沒有 `chapter`；同時非目標又禁止改第一章 runtime。三者不能同時成立。

**修正門檻**：

1. 定義跨章唯讀判定的精確函式與條件，例如只接受經第一章 `loadSave + sanitizeImport` 通過的 schema 3，並明定 `ended`／E3 子項何者構成可引用；壞檔一律安全降級。
2. 書信碼採版本化「系列信封」或維持每章 raw code，二者擇一；若採信封，須向後相容 legacy 第一章 raw code，並明定匯入時寫入哪個章鍵。
3. 加無存檔、未完章、合法完章、壞 JSON、錯 schema、legacy raw code、第二章 code 七條常駐測試。
4. 寫清楚第二章是通關後解鎖或可直接開始；不可把這個產品決定藏在 fallback 實作裡。

### A-4｜敵方數據卡的四步狀態機與凍結劇本 B3-5 順序不相容

**位置**：規格第 38–47 行；凍結劇本第 341–365 行。

規格要求辛普里奧先出數據卡，玩家判斜率、分類，正解後取得 F5 最後一環；但凍結劇本先讓玩家選 `over／honest`，只有 `over` 分支才出示遠砲數據，`honest` 會直接進 F5。照規格做會改劇本順序；照劇本做則誠實路線完全跳過 Sol verification 要求的玩家數據判讀。

另有三個未定狀態：判斜率錯誤是否扣分、分類正確後如何進 F5、分類錯誤歸零再入時是否重做兩步。

**修正門檻**：畫出 B3-5 的逐步狀態表（入口條件、可見卡、玩家動作、扣值、旗標、再入位置、F5 解鎖）。若決定把數據卡移到 `over／honest` 之前或讓分類取代原 A/B 選擇，必須對凍結劇本走一次正式變更紀錄；不可由 UI 私自重排。

### A-5｜`√` lint 的揭曉規則會把玩家自己的發現選項判成違規

**位置**：規格第 49–57 行；凍結劇本第 226–241 行。

規格寫「F2 律成立前，`√／平方根／開方` 禁止出現在台詞＋UI」，但玩家在 F2 成立前就必須看見並選擇「射程像是下落高度的開方」。若 lint 忠實執行，凍結劇本必紅；若默默白名單，封閉列舉又失去契約。

**修正門檻**：把揭曉邊界改成可驗證事件，而非模糊的「成立前／後」。建議新增封閉類別 `player-hypothesis`：該選項可在指定節點首次出現，玩家選取或以數值押中後寫入 `reveals.sqrt`，此後 NPC 才能回述。負向測試須覆蓋：提前 NPC、提前卡牌、提前 ARIA、錯節點偽裝 `player-hypothesis`、缺 `reveals` 全部必敗。

## 四、B 級｜需在 v0.1.1 寫清

### B-1｜正式章沒有承接工坊切片的玩家門檻

切片的 90 秒組裝、≥70% 能說出選擇如何改變數據、三誤差能診斷至少兩種、重玩換路徑、組裝時間低於章遊玩 20% 等門檻，在正式規格完全消失。正式章至少要保留可觀察的內部驗收；若此時無法場測，先列為候選版人工 checklist，不要只驗引擎矩陣。

### B-2｜換球必經缺少「比較守衛」與失敗說明

規格寫探索預選木球、學者自由配置，但沒有規定學者若同時改 H、桌沿、量法或校準狀態時如何拒絕。應明定：兩筆成立主張只允許球不同，其餘裝置指紋與發射校驗相同；錯選列出差異維度、不寫入 F2b、不耗掉斷言機會。

### B-3｜無第一章存檔時的 F5 降級鏈仍是占位描述

`S4+F2` 不是可直接實作的三槽答案。請列出有／無第一章基線兩套完整槽位、玩家可見文字、正解與干擾項；兩條都必須導向相同條件化結論，且無存檔版不能假裝玩家親手做過第一章實驗。

### B-4｜「無條件同時落地」不能只靠字串搜尋

`在這座機關聽得出的範圍內，兩球同時落地` 含有相同子字串，卻是合法條件句。請沿第一章封閉類別做語義標記（例如 `resolution-qualified`），驗證合法類別必有解析度／條件欄；裸字串、錯類別與缺條件必敗。月球豁免只在掃描範圍真的跨章時保留。

### B-5｜資源與節奏預算未定

第二章仍顯示天數與信譽，但 R-LAB2 沒有每次組裝、校準、試放、正式 run 的時間成本，也沒有紀錄上限／摺疊條件。請至少定義哪些動作耗天、玩家最少與合理最多需幾次 run，並沿用「紀錄不可刪，只能聚焦／摺疊」原則。

## 五、C 級｜機械同步與命名

### C-1｜測試基線已不是 65/65

規格第 73 行寫第一章 65/65；目前是 66/66。建議改為「提交時現行第一章 suite 全綠，禁止刪弱測試」，不要把會持續增加的數字寫成永久法條。

### C-2｜profile 名稱只可做內部 ID

`毛邊方向散／目測粗讀／未校準速度漂移` 已帶故障歸因。內部映射可這樣命名，但玩家 UI 只能先呈現可觀察特徵（散、粗、漂），再由筆記與伽利略問句漸進診斷；不得直接顯示 profile 名替玩家判案。
