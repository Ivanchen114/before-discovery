# 美術製作規格書 v0.1 — 技術可行性檢查

**執行者**:Claude(依 ADR-001 §8.2 之義務)|**日期**:2026-07-18
**結論**:**無 A 級阻擋。** 整體與靜態網頁架構(GitHub Pages、免建置、AI 可拆卸)高度相容;「預設不用全螢幕影片」「文字不燒進圖」「程式原生 UI」三條尤其正確,替效能、在地化與無障礙省下大量未來成本。以下為 B 級建議與 C 級備註,供 v0.2 修訂參考。

## B 級(建議納入修訂,各附替代方案)

**B-1|檔案量預算需要硬數字。** 每章 6–10 張背景+2–10 位角色×表情差分,runtime 壓縮後粗估每章 10–30MB。GitHub Pages 限制:單檔 100MB、repo 建議 <1GB、頻寬 100GB/月——技術上可行,但手機端體感取決於首屏量。§6.2「記錄下載量」是事後量測,建議改為事前預算:**每章 runtime ≤25MB、首屏 ≤3MB、最大單檔 ≤2MB**,超標即壓縮或砍變體。數字可於 Art Proof 後修訂,但欄位現在就該存在。

**B-2|token 需要單一真相來源。** §1.3 色彩 token 與 §5.6 動效 token 我會落地為 CSS custom properties;建議凍結一份 `tokens.json`(名稱+語意+暫定值),Style Bible 與程式共用同一檔,避免兩邊各自維護後漂移。此屬「技術接口」,凍結期允許,隨時可建。

**B-3|立繪差分的 runtime 合成方式未定,這是最貴的返工坑。** 選項:(a) 每表情一張整圖——實作最簡單,流量與維護成本最高;(b) 身體層+臉部差分疊層——省流量,但角色母版規格必須加「差分錨點座標」與裁切框欄位,且生成式影像產出的差分對齊需人工校正。建議 Art Proof 階段就用一位角色實測 (b) 的對齊工作量,再決定全案策略;若 (b) 成本過高,退 (a) 並靠 B-1 預算控管。

**B-4|「不直接載入母版」隱含一個資產縮壓步驟。** 與灰盒「無建置」不衝突(該條只約束灰盒),但正式版需要極簡管線(單支腳本產出多尺寸 WebP 即可,不必引入 bundler)。建議先立一條 ADR:遊戲本體永遠免建置,資產管線允許獨立腳本——邊界說清楚,日後不會被「順便上 Webpack」滑坡。

**B-5|lazy loading 與 file:// 的邊界。** `<img>` 在 file:// 下可用,fetch/JSON 不行(已由 GB-ADR-001 解決);但 §6.2「延遲載入」若用 IntersectionObserver+動態 img 可以雙棲,若用 fetch 資源清單則 file:// 會壞。建議規格明文:**資產清單一律走 .js 載體**(沿用灰盒策略),延遲載入用原生 `loading="lazy"` 或動態 img。

## C 級(備註,可忽略)

**C-1**:`prefers-reduced-motion` 與動效 token 設計相容,實作零風險;唯「所有動效可被替代」意味動效以 CSS class 驅動為原則,inline JS 動畫僅限實驗模擬本體。
**C-2**:SVG 圖表配 `<text>` 元素可同時滿足在地化與無障礙等價資訊,建議 Style Bible 直接規定圖表文字一律 SVG text/HTML,不入點陣。
**C-3**:§5.4 立繪長邊 2400px 母版合理;注意生成式工具原生輸出多為 1024–2048px,放大工作流(esrgan 類)需列入 §7 的紀錄欄位。
**C-4**:待決事項 4(字體):可商用開源中文字體實際選項不多——思源黑體/宋體、jf open 粉圓、芫荽等;嵌入網頁需 subset,每字重約 1–4MB,建議列入 B-1 的首屏預算計算。
**C-5**:G1 解鎖條件「灰盒通過」與開發凍結令一致,程式端無時程衝突;灰盒已可玩,Art Proof 的「程式動效 proof」(§4.1 第 6 項)屆時可直接在灰盒 repo 上做,不必另起爐灶。
