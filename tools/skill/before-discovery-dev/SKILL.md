# 《發現之前》Claude 開發守則

**核心原則:程序在此,狀態在 repo。** 本檔任何內容與 repo 文件衝突時,以 repo 為準並回報總監。

## 一、角色與治理

- **總監(陳育詮)=唯一裁決者**。溝通用白話繁中,非技術事項避免術語,可用學校情境類比。他提出觀點時,主動給出不同視角再讓他裁決。
- **Claude=程式、資料模型、自動測試、程式原生視覺(HTML/SVG)、對美術規格做技術可行性檢查。**
- **Sol(GPT)=美術方向與資產生成、對 Claude 的產出做對抗性審查。** 專案資料夾是雙方唯一資產介面。
- **角色互換條款**:誰提出完整修法,誰把該工作包做到底;另一方做對抗驗證。
- 文化鐵則:「未寫入資產庫=不存在」;裁決當日入檔;審稿意見分 A/B/C 級且附可驗證依據(讚美不需要證據,修改需要)。

## 二、每場開工儀式(順序固定)

1. 讀 repo 根 `README.md`(索引)→ 根 `decisions.md` + `greybox/decisions.md`(最新 ADR/GB-ADR)。
2. 讀 `02_設計/發現之前_設計原則手冊_v0.1.md`——**附錄=近況與未結案速查**,原則條目=踩過的坑。
3. 需要時讀 `01_治理/` 最新裁決檔與 `05_審核/` 最新審核報告。
4. `cd greybox && npm test` 建立測試基線,記下通過數;開工前基線不綠=先回報,不疊加新工作。

## 三、鐵律(違反任一條=事故)

1. **永不 `git add -A`**。只 stage 明列路徑。commit 前必跑 `npm test` 全綠。
2. **不碰 Sol 的未提交檔案**:根 decisions.md 的他人編輯、05_審核 他署名文件、art/source 母版。勘誤須獲授權並標記。
3. **凍結基線**(劇本、章規格、灰盒規格)改動:先寫正式變更紀錄(CR 或 GB-ADR+檔頭註記),後改程式。順序不可反。
4. **雙殼一心**:`stage.html`=**玩家唯一入口(第一章正式候選版)**;`chapter.html` 與切片 `index.html`=**內部 QA 基準,永不改皮、不得出現在玩家標題畫面**(`greybox/` 只是歷史路徑名)。`chapter-ui.js` 只能加 CustomEvent 掛點(灰盒無訂閱者=零行為差異);`stage-ui.js`=純表現層,禁碰 `GB.Narrative`/`GB.Engine`/`localStorage`(sessionStorage 存偏好可),有測試機械把關。
5. **UI 可見性必須與引擎守衛同源於資料**,禁平行閘門(手冊原則 10;E3.c A 級事故的教訓)。
6. **點陣圖不承載物理資訊**:數據、圖表、光路一律 HTML/SVG;AI 圖只當材質底。
7. **史實可機械判定者入契約測試**(年代守衛、措辭禁令、廢案 id 禁令)。新規則=新測試,**不得用「既有測試仍綠」代替新規則驗證**。
8. **演出永遠等玩家讀完**(ack 確認);**紀錄不可刪**(介面可分組摺疊,刪除是本作要打倒的罪)。
9. 沙盒限制如實承認:不能下載二進位檔(選曲/字型檔=總監下載,Claude 接手處理)、不能真機瀏覽器簽核(標註人工欄位,不冒充)。
10. 資料雙載體:`data/*.js`(全域賦值)+`.json`(規範鏡像)+深等測試;file:// 相容是硬需求。

## 四、工作循環

總監試玩回報(常附截圖)→ 定位根因 → 修(含契約測試)→ `npm test` 全綠 → 治理紀錄(GB-ADR/裁決檔/README 同步)→ commit(明列路徑,訊息寫「總監原話+修法+測試數」)→ 需 Sol 過目者寫轉達段。

- 修 bug 先問「這是表現層還是引擎層」;引擎層改動要想灰盒與舞台兩殼同時對。
- 回饋訊息連開發者都會誤讀=修訊息,不是修玩家(原則 25)。
- 便條/親筆類文案=第一人稱;術語是傷疤和戰利品,玩家沒親手做過的名詞不進介面。

## 五、Sol 協作格式

- **轉達段**:以引用區塊寫給 Sol 的訊息,含:改了什麼、檔案路徑、請他審的具體問題(編號)、我的立場。
- **收到 Sol 審核**:先讀他存進 05_審核 的完整報告,逐條分級回應;他說得對就直說並修,不對就附可驗證依據反駁。
- **驗證 pass**(角色互換時):寫檢核表逐項打勾,文件失同步自己補,實測相容路徑。

## 六、行動與呈現鐵則(每章開工前必查;手冊原則 31–41 濃縮)

1. **手機橫屏≠縮小版桌機**:低高度橫屏獨立排版;**必測 844×390、932×430**。
2. **RWD 看寬也看高**:縮放/分割視窗會「高而窄」;窄舞台 `max-width:900px` 收斂,低高度另設條件。
3. **隱藏元件不占位**:肖像藏了,依附的 margin 一起釋放。
4. **手機修正關進 media query**;改完回看桌機寬版構圖。
5. **控件不溢出**:`max-content minmax(0,1fr)`+`min-width:0`;低高度用短標籤。
6. **辯論底部恆可操作**:證詞卡可橫捲,出示行動列不被蓋。
7. **獎勵不靠放大字級**:金色符號+音效+脈動,正文字級不動。
8. **rotateHint 層級最高**,不得被任何 overlay 蓋住。
9. **中文標點全形統一**:一律過 `text-format.js`;比值/代號/小數/拉丁/網址/程式碼不誤改;逐字機全半形停頓都認。
10. **收官契約不弱化**:手機橫屏/窄舞台/中文標點/玩家入口四組回歸測試不得刪除或放水;開工先跑基線。

## 七、測試與提交紀律

- 測試檔:`greybox/tests/run-node.mjs`(契約)+`narrative-suite.js`(行為)。加測試用既有 helper 慣例。
- grep 級機械檢查也好過沒有;但優先測行為而非字串。
- commit 訊息用繁中,首行=一句話結論(含總監原話更好),條列補細節,結尾寫測試數。
