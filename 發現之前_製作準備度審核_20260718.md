## 二、A 級發現（阻擋定稿）

### A-1｜附錄 A 數值違反通過矩陣，共 12 處（R-EXP-04 / R-DATA-03）

規格書上游問題清單自我標記 2 格可疑（陡+水鐘、緩+脈搏三平均）。本審核依 R-DATA-01 模型與 R-JUD-03 判定規則全面驗算，實況更嚴重：

| 矩陣格 | 規範 | 實測 |
|---|---|---|
| 陡+水鐘 | 單次、三平均皆不通過 | **三個 run 單次全通過，三平均也通過** |
| 原木槽面（全部 9 組合） | 一律不通過 | **7 組違規**；三平均幾乎全過，多數 run1 單次即過 |
| 木球 | 一律不通過 | **中+水鐘、陡+水鐘、陡+音格 3 組違規**（run1/run2 單次通過） |
| 緩+脈搏 | 單次不過、三平均過 | 相符（規格擔心的這格其實沒事） |

示例：陡+水鐘 run1 讀值 (22, 71, 105, 159)，對 1:3:5:7 的偏差僅 7.6%／4.5%／3.2%，遠在 12% 容差內——玩家第一次就能判定，而設計意圖是「計時型誤差隨傾角加劇、應不可判定」。

**底層原因是量綱錯配**：樣式偏移是「絕對刻度值」，判定容差是「相對百分比」。傾角越陡基準越大（陡 Δ1=22 vs 緩 Δ1=12），同樣 ±5 的絕對偏移相對縮水，於是一切高基準配置都變得可判定。原木樣式另有一個獨立問題：四段偏移近似等比例（約 −8%～−14%），等比例衰減不破壞比值結構，摩擦「型」反而保住了奇數律的「形」。

**修正方向（三擇一或混用，屬作者職權）**：①樣式改為基準的百分比（乘法噪音）；②樣式表依傾角縮放；③容差依傾角分級。無論選哪條，修訂後以附件腳本重跑一次即知過不過——建議此腳本直接納入 `/tests` 作為 R-DATA-03 的第一個實作。

**可重現依據**：`附件_通過矩陣驗算_20260718.py`（本資料夾），python3 直接執行。

判定口徑說明：「單次通過」以三個預設 run 全數可判定為準、「不通過」以無任何 run 可判定為準。因 R-EXP-03 是確定性輪替，run1 可判定＝首次玩家必得可判定數據，故「部分 run 通過」對「不通過」格同樣構成違規。

**B-6｜劇本-規格對齊債**：劇本 E3 未分 a/b/c 子項（規格上游清單已報）；劇本場景 2-3 的數據是理想值 1-4-9-16，與帶噪 fixture 敘事不符。依解凍條件，劇本重修排在灰盒通過之後即可，先入債務清單。

**B-7｜切片外規格增補**：序幕、高塔、P1/P2、E4/E5、存檔、模式分層均「定崗後另行增補」——這是第一章灰盒完工的前置，別漏排。

**B-8｜時程與人力未估**（架構書待決事項 5）。零行程式的專案沒有排程基準，建議 bake-off 本身限時（例如 1–2 週），用實測工時回推第一章與全作量級——這正是灰盒的另一個功能。

## 五、風險提示（C 級與公司視角）

**C-1｜遙測矛盾**：v1 驗收標準是「公開市場反應」，但 v1 承諾零資料蒐集。不裝任何量測，市場反應用什麼判讀？建議 v1 前定義代理指標（itch.io 頁面數據、GitHub stars、自願回饋表單），或明文接受純質化評估。這是驗收標準與技術承諾的直接衝突，遲早要裁決。

**C-2｜行動裝置缺口**：灰盒排除行動裝置合理，但目標受眾是國中～高中生——他們的「網頁」九成是手機。桌面 only 的 v1 會嚴重壓縮觸達。至遲於 v1 前補回路線圖。

**C-3｜文言證詞 vs 國中生底線**：辛普里奧的「千載不刊之論」「重物性喜就下」很有味道，但設計底線是國中生可完整遊玩。建議在試玩觀察協定加一條：記錄閱讀理解卡點（目前協定只記操作卡點）。

**C-4｜「六門檻」無定義**：流程 §6.8 解凍條件寫「通過六門檻」，全套文件無此詞定義。推測＝灰盒四問＋兩題通關訪談，建議明文化，否則解凍時必起爭議。

**C-5｜版次治理**：資料夾平鋪三版架構書。建議建 `archive/` 收舊版，現行版唯一露出——B-2 的引用飄移就是這個風險的實例。

**C-6｜Bus factor**：單人總監＋雙模型。流程設計已盡量把知識寫進資產，維持此紀律即是唯一解，無需額外動作。
