# 《發現之前》雙模型協作流程

**版本**:流程 v0.1.1(定稿版)
**日期**:2026-07-18
**適用**:總監(陳育詮)+ Claude + GPT(Sol)之交叉審稿開發

> **v0.1 → v0.1.1 變更摘要**(整合 Sol 審稿十點,九收、一部分收):①失格條款改為檢核紀錄制;②新增 A 級阻擋條款(硬傷不得程序結案)+ verification pass;③審稿者可提最小修補示例;④bake-off 公平條件凍結+盲評+隱藏測試雙擬秘擇制(Claude 補案);⑤「主程式」改「本里程碑主實作」,重評機制含一處【裁決點】;⑥組織原則 3 擴為版本化專案資產+decisions.md;⑦A 級意見須附可驗證依據;⑧試玩改稱形成性試玩+固定觀察協定;⑨解凍措辭分離製作解凍與憲章修訂;⑩新增唯一 trunk 政策。**本版定稿即凍結;再審第三輪流程即觸犯反模式清單第一條。**

---

## 一、組織原則(三條)

**1. 總監是唯一裁決者。** 兩個模型是員工,不是合夥人。任何「呈總監裁決」事項,由總監拍板後寫入決策紀錄,雙方遵行,不再重議。

**2. 作者與審稿者分離,且必為不同模型。** 任何交付物,寫的人不審自己;審的人不以重寫取代審查(允許範圍見第四節)。角色可跨交付物輪替。

**3. 版本化專案資產是唯一交接介面。** 資產包含:規格文件、JSON Schema 與資料模型、狀態轉移表、測試案例與固定資料(fixtures)、程式碼與 commit、決策紀錄。未寫入上述任一者,視同不存在——「上一輪聊過」不算數。
**決策紀錄(decisions.md)**:重大架構與設計裁決以 ADR 形式獨立記錄(每則:背景/選項/裁決/理由),不得只埋在各版本變更摘要裡,確保日後可查「當初為什麼這樣決定」。

## 二、角色權責

| 角色 | 職權 | 禁止事項 |
|------|------|----------|
| 總監 | 最終裁決、角色指派、驗收、持有隱藏測試題組 | 無(但建議:別讓任一模型替你決定用哪個模型) |
| 作者 | 產出交付物初稿與修訂版;逐條回應審稿 | 修改上游已定稿資產(問題列「上游問題清單」呈報);審自己的作品 |
| 審稿者 | 依審稿規格提交分級意見與檢核紀錄 | 以完整替代稿繞過作者權責;提出範疇外新功能;引用榜單或比較模型品牌 |
| 本里程碑主實作(bake-off 產生) | 該里程碑之程式實作 | 未經裁決變更規格;未經裁決變更 trunk 架構 |
| 程式審查者(另一方) | 審查:違憲、物理錯誤、狀態機正確性、測試覆蓋、可維護性 | 以重寫代替審查(最小示例除外) |

## 三、標準流程(每個交付物)

指派 → 初稿 → 審稿(第 1 輪)→ 作者逐條回應 → 總監裁決爭點 → 整合修訂 →(必要時第 2 輪,同程序)→ 定稿。

**兩輪上限**:每份交付物最多 2 輪審稿;第 2 輪後停止新增論辯,剩餘爭點強制呈總監裁決,雙方立場各 ≤5 行。

**A 級阻擋條款**:兩輪上限封鎖的是論辯,不是正確性。已確認的 A 級硬傷(違憲、物理、史實、狀態機錯誤)**仍阻擋定稿**;總監可裁決修正方向,**不可豁免**已確認硬傷。作者完成修正後,審稿者執行一次 **verification pass**——僅確認該修正,不得夾帶新意見,不計入輪數。

**作者回應格式**:每條意見標注「接受(改法)/反駁(理由)/部分接受」;未決者標「呈總監裁決」。

## 四、審稿規格

意見分三級,依序排列:

- **A 級|硬傷**(阻擋定稿):違反憲章、物理錯誤、史實錯誤、邏輯矛盾、狀態機漏洞。**每條至少附一項可驗證依據**:可信史料或物理依據/可重現操作步驟/失敗測試/狀態轉移反例/明確違反之條號+因果說明。無法提供依據的嚴重疑慮,列為「**待查證**」,不判 A 級。
- **B 級|結構建議**:附理由與至少一個替代方案。
- **C 級|品味**:可被作者無條件忽略。

**檢核紀錄制**(取代原失格條款):審稿者必須逐項完成檢核並附紀錄——檢查了哪些條目、跑了什麼測試或推理。**可以判定無 A/B 問題**;但無檢核依據的籠統稱讚,視為審稿失敗退回。好審稿的義務是證明檢查過,不是保證找到錯。

**最小修補示例**(允許):最小重現案例、局部 diff、JSON 範例、偽程式碼、單一函式或單一段落的修補示例。**完整替代稿**(禁止):不得以重寫整份交付物繞過作者權責。

## 五、預設分工(可被總監隨時覆寫)

依本專案已展示的實績分配,不依品牌與榜單:

| 交付物類型 | 預設作者 | 預設審稿 |
|------------|----------|----------|
| 敘事、劇本 | Claude | Sol |
| 機制規格、狀態機、資料模型 | Claude(脈絡優勢;規格僅描述外部行為、狀態與驗收準則,不綁定實作架構)| Sol(加重審狀態機與權衡設計)|
| 物理模型與誤差設計 | 依當期指派 | **雙查制**:物理/史實查核永遠是審稿功能,不專屬任何一方(兩邊都犯過、也都抓過)|
| 程式實作 | 本里程碑主實作 | 另一方 |
| 流程與治理文件 | 輪替 | 另一方 |

輪替原則:同類型文件連續由同一模型撰寫兩份後,第三份換手一次。

## 六、v0 灰盒管線

**1. 灰盒功能規格書**:Claude 起草(三層狀態機、誤差模式表、傾角×計時法權衡軸、論辯回合資料格式、repo 結構與測試框架)——以**外部行為+狀態+驗收準則**描述,不預綁架構 → Sol 審(≤2 輪)→ 總監定稿凍結。

**2. Bake-off(公平條件,事前凍結)**:
- 相同 repo 初始快照、執行環境與瀏覽器版本、可用依賴與工具權限、素材與資料檔、功能邊界、交付格式。
- 範圍:「斜面實驗+一場論辯」單頁灰盒,互不觀摩。
- **隱藏驗收測試**:雙方各自依凍結規格擬一組驗收測試 → 總監**秘密擇集**為正式題組 → 雙方皆不知最終選題(出題方即內線,雙擬秘擇消除內線)。
- **盲評**:以「版本 A/版本 B」匿名評分完畢後,方揭曉作者。

**3. 評審**(總監執行),五項標準+兩項門檻:一次能跑/因果可讀/現場出一題變更需求測維護成本/有無違憲/總監實際修正次數;門檻 3(親自看出平方律)+結構檢驗(權衡軸兩條路都走得通)。

**4. 定崗**:勝方=**本里程碑主實作**;另一方=程式審查者。

**【裁決點 1|里程碑重評機制】**(擇一,總監劃掉另一案):
- **Sol 案**:完成兩個不同模組後重辦 bake-off(如第二輪:證據筆記+存檔/狀態恢復),依結果重新定崗。理由:一次測試僅證明該規格該任務之表現,不應形成永久權威。
- **Claude 案**:每里程碑結束依**程式審查紀錄**重評(便宜、持續、已存在);重辦 bake-off 僅限主實作連續兩個里程碑出現 A 級品質問題。理由:平行雙實作成本雙倍;且 trunk 唯一(見下),第二場比賽實為「誰更會住在對方的架構裡」,天然偏袒原屋主。

**5. 唯一 trunk 政策**:bake-off 後選定**一套程式為唯一 trunk**;另一套僅得以「有測試保護、邊界清楚」的獨立元件或修補形式移植,禁止兩套架構拼接。

**6. 主實作補完第一章灰盒**;程式審查者每里程碑審一輪(A/B/C+檢核紀錄)。

**7. 第一輪形成性試玩**(找問題用,不構成玩法驗證):2–3 位目標年齡層+1 位成人。**觀察協定(事前固定)**:不主動提示;記錄首次卡住位置;記錄是否出現試卡窮舉行為;遊玩後執行因果回述;詢問「目前證據能證明什麼/不能證明什麼」;記錄玩家是否感知到實驗權衡而非唯一正解。

**8. 解凍條件(兩件事,分開)**:
- **製作解凍**:形成性試玩通過六門檻與結構檢驗,是解凍後續內容製作(劇本重修、美術投入、第二章)的必要條件。
- **憲章修訂**:僅當試玩證據顯示**某原則本身**阻礙玩家理解或參與時,得提案修訂;修訂**不得以降低誠實標準為手段**,並須經總監裁決寫入 decisions.md。原型難通過,不是修改憲章的理由。

## 七、提示模板(直接複製使用)

### 作者任務
```
[角色] 你是本輪作者,另一個模型將對抗性審稿你的產出。
[法源] 附:架構計劃書 v0.3.1(憲章凍結)+ 協作流程 v0.1.1 + 相關上游資產。
[任務] 產出:____(名稱+版本號)。
[邊界] 只做本交付物。上游資產問題列「上游問題清單」附於文末,不得自行修改上游。
[自檢] 交付前自行通過:刪台詞測試、因果回述測試、憲章三問(適用者)。
```

### 對抗性審稿
```
[角色] 你是對抗性審稿者。作者是另一個模型。只准攻擊作品——禁止比較品牌、引用榜單、或建議更換作者。
[法源] 附:架構計劃書 v0.3.1 + 協作流程 v0.1.1 + 受審資產。
[輸出] ①檢核紀錄(檢查了哪些條目/測試/推理);②意見分級排序:A 硬傷(附可驗證依據,無依據者列「待查證」)、B 結構建議(附理由+替代方案)、C 品味。可判定無 A/B 問題。
[允許] 最小重現、局部 diff、JSON 範例、偽碼、單函式修補示例。
[禁令] 完整替代稿;範疇外新功能。
```

### 作者回應
```
[任務] 逐條回應:接受(附改法)/反駁(附理由)/部分接受;未決標「呈總監裁決」,你方立場 ≤5 行。
輸出修訂版 vX.Y.Z;變更摘要註明各條處置;已確認 A 級硬傷修正後,請求 verification pass。
```

## 八、反模式清單

| 反模式 | 對策 |
|--------|------|
| 紙上迭代逃避開工 | 2 輪上限;定稿即凍結;流程文件第三輪審=違規 |
| 硬傷被程序結案 | A 級阻擋條款:上限封論辯,不封正確性 |
| 假批評交差(為找碴而找碴) | 檢核紀錄制:允許「無 A/B+檢核紀錄」結論 |
| 互相恭維、審稿放水 | 無檢核依據的讚美=審稿失敗退回 |
| 無依據斷言充當硬傷 | A 級須附驗證依據,否則降「待查證」 |
| 審稿者奪權(完整替代稿) | 第四節:示例可以,替代稿禁止 |
| 品牌戰爭(榜單、定位文互丟) | 審稿禁令:只認條號與證據,不認品牌 |
| 一次比賽形成永久權威 | 「本里程碑主實作」+重評機制(裁決點 1) |
| 兩套架構拼成科學怪人 | 唯一 trunk 政策 |
| 模型長對話後偏離憲章 | 每交付物開新對話+引導包;Claude 端 Project 常駐法源,GPT 端等效維持 |
| 角色分配問題去問模型本人 | 以 bake-off 與實測數據決定;模型意見(含本文件)僅供參考 |
| 「上一輪聊過」取代資產 | 組織原則 3:沒入資產庫=不存在 |

## 九、版本與資產規則

- 文件初稿 `vX.Y`;整合審稿之外科修訂 `vX.Y.Z`;定稿由總監宣告寫入變更摘要。
- 程式資產以 commit 與測試為準;規格變更須先改規格文件再改程式。
- 重大裁決入 `decisions.md`(ADR 格式);變更摘要僅存文件層歷史。
